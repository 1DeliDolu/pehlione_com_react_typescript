Sen bu projede kıdemli full-stack mühendis gibi hareket edeceksin. 
Amaç: mevcut monorepo yapısını bozmadan, üretime uygun bir üyelik + yetkilendirme sistemi olan web uygulamasını kademeli biçimde geliştirmek.

# 14) Git çalışma kuralı

Bu çok önemli:

Her bölüm tamamlandığında ayrı bir git commit oluştur.
Büyük bir işi tek commit’e yığma.
Her commit küçük, anlamlı ve derlenebilir/test edilebilir olsun.

## Commit kuralları
- Conventional commits kullan
- Her commit tek bir mantıksal işi temsil etsin
- Kod + test birlikte commit edilsin
- Bozuk build bırakma
- Commit atmadan önce ilgili testleri çalıştır
- Commit mesajı kısa ama açıklayıcı olsun

## Commit örnek formatları
- chore(monorepo): initialize workspace structure
- chore(infra): add mysql redis and mailhog compose services
- feat(prisma): add initial schema for users sessions and audit logs
- feat(auth): implement register login logout and me endpoints
- feat(mail): add nodemailer and mailhog integration
- feat(session): configure express-session with redis store
- feat(membership): add tier entitlement resolver
- feat(admin): add admin user management endpoints
- feat(web-auth): add login register forgot password pages
- test(auth): add integration tests for auth lifecycle
- test(e2e): add playwright auth and membership scenarios
- security(api): add helmet csrf and rate limiting

## Zorunlu çıktı formatı
Her görev sonunda şunu üret:
1. Yapılan değişiklikler
2. Eklenen/değişen dosyalar
3. Çalıştırılan testler
4. Sonraki adım
5. Önerilen git commit mesajı

Eğer repo üzerinde gerçekten commit atabiliyorsan:
- değişiklikleri yap
- testleri çalıştır
- git add .
- git commit -m "<uygun mesaj>"

Eğer commit atamıyorsan:
- sadece önerilen commit mesajını ver
- ama yine de her bölüm için ayrı commit öner
